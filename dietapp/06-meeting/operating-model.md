# 공동 운영 모델 (Co-Founder Operating Model)

공동창업이 망하는 90%는 **운영 모델 부재**에서 시작. 미팅에서 이걸 명시하면 시니어니스 신호.

---

## 1. 역할 분담 (Day 1 기본형)

| 영역 | 본인 (CEO/Operator) | 파트너 (Chair/Sponsor) |
|---|---|---|
| 제품 전략 | Lead | Consult |
| 채용 (시니어+) | Lead | Approve |
| 채용 (주니어) | Lead | Inform |
| 마케팅 예산 (≤₹3L/월) | Lead | Inform |
| 마케팅 예산 (>₹3L/월) | Recommend | Approve |
| 가격 변경 | Recommend | Approve |
| 신규 시장 진입 | Recommend | Approve |
| 자본 조달 (Series A 등) | Lead | Approve |
| Exit 의사결정 | Joint | Joint |
| 법무·회계 | Lead | Inform |
| 도메인 협업 (의사·영양사 네트워크) | Consult | Lead (있다면) |

**원칙:**
- 일상 운영 = 본인 단독 결정
- ₹3L+ 또는 회사 방향성 = 합의
- 본인이 모든 의사결정 단독 못하면 운영 마비, 파트너가 다 결정하려 하면 본인 효용 0

---

## 2. 의사결정 4단계

| Level | 정의 | 의결 |
|---|---|---|
| L1 | 일상 (≤₹50K, 1주 영향) | 본인 단독 |
| L2 | 분기 영향 (≤₹3L) | 본인 단독, 파트너 inform |
| L3 | 회사 방향 (≥₹3L 또는 12개월 영향) | 본인 + 파트너 합의 |
| L4 | 회사 운명 (펀딩, 피봇, exit) | 만장일치 |

**합의 안 되면:**
- L3: 30일 cooling period → 재논의 → 그래도 안 되면 본인 단독 결정 권한 (단 분기 1회 한도)
- L4: 외부 advisor 1명 의견 청취 → 그래도 합의 안 되면 status quo 유지

---

## 3. 운영 cadence

### 일일
- Slack 비동기 standup (어제·오늘·blocker)
- 본인 + 개발자 + 디자이너만, 파트너는 read-only

### 주간 (모두 1시간)
- **월 11:00 — Product** (본인 + Tech + Design + RD)
- **수 14:00 — Growth** (본인 + Marketing + 컨텐츠)
- **금 17:00 — Co-founder 1:1** (본인 + 파트너) ← **이게 가장 중요**

### 월간 (3시간)
- 첫째 월요일: KPI 12종 review + 다음 달 OKR
- 파트너 + 핵심 인력 4~5명 참석

### 분기 (반나절)
- 분기 retrospective + 다음 분기 OKR
- 외부 advisor 1명 동석

---

## 4. Co-founder 1:1 운영 규칙 (금 17:00)

**Agenda 고정 (60분):**
1. (10분) 지난주 KPI 5종 review
2. (15분) 본인의 win 3 / learn 3
3. (15분) 파트너의 win 3 / learn 3
4. (10분) 다음 주 우선순위 합의
5. (10분) "지난주 너에게 답답했던 거 1개" — 솔직 피드백 의무 슬롯

**규칙:**
- 절대 cancel 금지 (출장 시 zoom)
- 메모 공유 (Notion 공동 페이지)
- 4번째 항목 ("답답했던 거")은 **건너뛰면 위험 신호** — 6주 연속 0개면 advisor 콜

---

## 5. KPI dashboard 공유

12종 KPI 모두 파트너에게 read-only 접근 (Mixpanel + Metabase 링크).

| KPI | 업데이트 주기 | 알림 트리거 |
|---|---|---|
| 신규 가입 | 일 | 일 평균 -20% 시 본인 + 파트너 동시 알림 |
| Free→Paid 전환 | 주 | 월 단위 -30% 시 알림 |
| MRR | 일 | 월 m/m -10% 시 비상 |
| Cash position | 주 | runway <6개월 시 비상 |
| D7/D30 retention | 주 cohort | 코호트 -5%p 시 알림 |

**투명성 = 신뢰**. 파트너가 숫자에 접근 못 하면 6개월 안에 관계 붕괴.

---

## 6. 자본 운용 권한

### 본인 단독 운용 가능
- 월 OpEx 예산 (사전 합의된 분기 plan 내) 자유 집행
- 채용 (사전 합의된 headcount plan 내)
- 인프라·SaaS 결제

### 파트너 사인 필요
- 합의된 분기 plan 초과 지출
- 자본 조달 (펀딩 라운드)
- 인수·매각·합병
- 분기 plan 자체 (분기 시작 시 합의)

### 은행 계좌 운영
- 회사 계좌: 본인 단독 결제권 ₹50K 이하, 그 초과는 본인 + 파트너 dual signature
- 또는 신용카드: 본인 한도 ₹2L/월

---

## 7. 정보 공유 의무

### 파트너 → 본인
- 도메인 네트워크에서 들은 시장 정보 (월 1회 이상)
- 본인 일정·재정 변동 (개인 사정으로 commitment 약화 시)

### 본인 → 파트너
- 모든 hire 의사결정 사전 inform
- 사용자 인터뷰 raw note (week 5건, Notion 공유)
- 매월 5일 서면 update (founder note 4페이지)
- 비상 상황 (key hire 이탈, WhatsApp 정지, 사용자 의학 클레임) → 24h 내

---

## 8. 분쟁 해결

### 우선 순위
1. Co-founder 1:1에서 솔직히 (Week 1)
2. Advisor 1명 의견 (Week 2)
3. 30일 cooling period
4. 그래도 안 되면 partnership-terms.md의 buyout 조항 발동

### 절대 하지 말 것
- 직원 앞에서 분쟁 노출
- Slack 채널에서 감정적 대화
- 파트너 동의 없이 외부에 회사 사정 공유

---

## 9. Onboarding (새 멤버 합류 시)

- 새 hire에게 본인이 day 1 설명: "회사 운영 구조 = CEO 본인 + Chair 파트너"
- 파트너 만나기 = 첫 주 안에 30분 1:1 (정기 X, but 오리엔테이션은 의무)
- 의사결정 라인 분명히: "L1·L2는 나에게, L3는 우리 둘이 같이 결정한다"

---

## 10. 운영 모델 review

- **3개월차** 운영 모델 자체 review (이 문서를 같이 다시 읽고 수정)
- **Series A 전후** 본격 board 거버넌스로 전환
- **연 1회** 외부 advisor가 facilitate하는 partnership health check

---

## 11. 자주 깨지는 지점 (선제 방어)

| 깨지는 패턴 | 방어책 |
|---|---|
| 파트너가 일상 결정에 일일이 개입 | L1·L2는 본인 단독 명시 (이 문서 그대로) |
| 본인이 큰 결정을 단독 진행 | L3·L4 합의 의무 명시 |
| KPI 안 보고 감정으로 다툼 | dashboard 공유 + 주간 1:1 fixed agenda |
| Commitment 비대칭 (한쪽만 풀타임) | partnership-terms.md vesting + reverse vesting |
| 신규 hire가 누구 따를지 모름 | onboarding day 1 명시 |
| 파트너 도메인 네트워크 소진 | 월 1회 정보 공유 의무 |
| 분쟁 시 advisor 부재 | Day 1에 advisor 1명 confirm |
